# Отчёт по проекту: IRF Kriging Forecasting Service

## 1. Постановка задачи

Цель проекта — разработать сквозную ИИ-систему для прогнозирования одномерного временного ряда с оценкой неопределённости. В качестве предметной области выбран прогноз месячных чисел солнечных пятен по датасету SILSO Monthly Mean Total Sunspot Number V2.0.

Проект решает задачу регрессии/прогнозирования временного ряда: по историческим значениям солнечной активности необходимо спрогнозировать будущие значения и оценить неопределённость прогноза.

Потенциальный пользователь системы — аналитик, исследователь или инженер, которому нужно:

- получить прогноз временного ряда;
- сравнить качество модели с простыми baseline-подходами;
- использовать модель через API, а не только через notebook;
- получить оценку неопределённости прогноза.

## 3. Разбиение на train/test

Использовано time-based разбиение без перемешивания, что корректно для временного ряда.

Текущая конфигурация эксперимента:

| Параметр | Значение |
|---|---:|
| Start year | 1970 |
| Total observations | 676 |
| Train all | 540 |
| Train used | 180 |
| Test | 136 |
| Season period | 132 месяцев |

Для ускорения обучения использовались последние 180 точек обучающей части (`max_train_points=180`). Тестовая часть содержит 136 точек.

## 4. Основная модель

Основная модель — `IRFKriging`.

Используемый kernel:

```text
RationalQuadratic + Periodic(T=season_period) + Nugget
```

Конфигурация модели:

```python
model = IRFKriging(
    SumKernels([
        RationalQuadratic(ps_bounds=(0.2, 100.0), len_scal_bounds=(3.0, 60.0)),
        Periodic(T=float(season_period)),
        Nugget(is_noise=True),
    ]),
    k=0,
    period=None,
    exp_term=[],
    exp_periodic_terms=[],
    optim_type="grad",
    n_starts=10,
    maxiter=50,
    warm_start=True,
    popsize=12,
    maxiter_de=15,
)
```

Модель возвращает:

- `predicted_mean` — прогнозное значение;
- `predicted_variance` — оценку дисперсии прогноза.

Обученная модель сохраняется в:

```text
project/artifacts/model.pkl
```

Метрики финальной модели сохраняются в:

```text
project/artifacts/metrics.json
```

## 5. Baseline-модели

Для проверки осмысленности финальной модели были реализованы простые baseline-подходы:

| Модель | Идея |
|---|---|
| NaiveLast | все будущие значения равны последнему train-наблюдению |
| SeasonalNaive | прогноз равен значению с сезонным лагом |
| MovingAverage | прогноз равен среднему по последнему окну |
| PolynomialRegression | полиномиальный тренд |
| IRFKriging | финальная модель |

Скрипт сравнения:

```text
project/src/models/evaluate_baselines.py
```

Результаты сохраняются в:

```text
project/artifacts/baseline_metrics.csv
project/data/processed/baseline_predictions.csv
```

## 6. Метрики качества

Использовались следующие метрики:

- `MAE` — средняя абсолютная ошибка;
- `RMSE` — корень из средней квадратичной ошибки;
- `MAPE` — средняя абсолютная процентная ошибка;
- `sMAPE` — симметричная процентная ошибка.

Для этого датасета `MAPE` плохо интерпретируется, потому что фактические значения солнечной активности иногда близки к нулю. Поэтому в выводах основной акцент сделан на `MAE`, `RMSE` и `sMAPE`.

## 7. Результаты экспериментов

Таблица baseline-сравнения:

| Model | MAE | RMSE | MAPE, % | sMAPE, % |
|---|---:|---:|---:|---:|
| IRFKriging | 21.735 | 27.244 | 653.638 | 57.644 |
| SeasonalNaive | 19.477 | 29.096 | 94.524 | 50.378 |
| MovingAverage | 62.592 | 71.978 | 2326.746 | 91.607 |
| NaiveLast | 64.342 | 74.228 | 2399.110 | 92.225 |
| PolynomialRegression | 537.799 | 609.039 | 10370.214 | 162.220 |

Выводы:

1. `IRFKriging` показал лучший `RMSE` среди рассмотренных моделей.
2. `SeasonalNaive` оказался сильным baseline по `MAE` и `sMAPE`, что ожидаемо для солнечной активности из-за выраженной цикличности.
3. `MovingAverage` и `NaiveLast` существенно проигрывают по RMSE.
4. Простая `PolynomialRegression` плохо подходит для этого ряда, так как не учитывает цикличность и сложную динамику.
5. IRF Kriging полезен тем, что возвращает не только прогноз, но и дисперсию прогноза.

## 8. API-сервис

Для интеграции модели реализован FastAPI-сервис.

Код сервиса:

```text
project/src/service/main.py
project/src/service/schemas.py
project/src/service/model_loader.py
```

Endpoints:

| Method | Endpoint | Назначение |
|---|---|---|
| GET | `/health` | Проверка состояния сервиса |
| GET | `/model/info` | Информация о модели и метриках |
| POST | `/predict` | Получение прогноза |
| GET | `/metrics` | Базовые технические метрики |

Пример запроса:

```json
{
  "horizon": 24
}
```

Пример запроса с пользовательской сеткой:

```json
{
  "grid": [540, 541, 542, 543, 544]
}
```

Пример ответа:

```json
{
  "grid": [540.0, 541.0, 542.0],
  "predicted_mean": [101.67, 98.80, 95.49],
  "predicted_variance": [181.74, 283.81, 394.91],
  "model_version": "irf-kriging-v1"
}
```

## 9. Валидация входа и обработка ошибок

В API добавлена валидация:

- `horizon` должен быть от 1 до 1000;
- `grid` не должен быть пустым;
- `grid` должен содержать только конечные числовые значения;
- слишком длинный `grid` отклоняется;
- при отсутствии `model.pkl` сервис возвращает ошибку `MODEL_NOT_FOUND`.

## 13. Безопасность и конфигурация

В проекте не используются внешние секреты или API-ключи. При этом соблюдены базовые практики:

- секреты не хранятся в коде;
- данные и артефакты отделены от исходного кода;
- входные данные API валидируются;
- ограничен максимальный размер `grid`;
- Docker-контейнер запускает только API-сервис.

## 14. Ограничения

Текущие ограничения проекта:

1. Используется один датасет и один основной временной ряд.
2. Подбор гиперпараметров ограничен.
3. MLflow или другой полноценный experiment tracking пока не подключён.
4. Наблюдаемость базовая, без Prometheus/Grafana.
5. API не реализует обучение на пользовательских данных, только прогноз по заранее обученной модели.
6. Визуализация прогнозов пока сохраняется таблично, без автоматического построения графиков.
